# PYDANTIC & INSTRUCTOR

---

Расскажи мне про инструменты Pydantic и Instructor для формальной логики и проверки текстов

---

Если говорить прямо, то Pydantic и Instructor хорошо подходят не столько для “формальной логики” в строгом академическом смысле, сколько для формализации требований к тексту и проверки того, что LLM или человек выдал текст в нужной структуре, с нужными ограничениями и без части типичных ошибок. Pydantic дает схему, типы, строгую валидацию, кастомные валидаторы, JSON Schema, strict/lax режимы и даже валидацию при изменении объекта через validate_assignment; кроме моделей, у него есть TypeAdapter для валидации по обычным Python-типам.  ￼

Для задач “проверки текстов” Pydantic полезен там, где текст можно разложить на явные поля и инварианты. Например: “в тексте должны быть тезис, аргументы, вывод, список рисков”; “оценка уверенности должна быть числом от 0 до 1”; “цитаты обязательны”; “запрещены пустые секции”; “дата должна быть валидной”; “список утверждений не должен быть пустым”. Это уже не свободный текст, а контролируемая структура, которую можно проверять field_validator и model_validator. Pydantic особенно силен именно в этой зоне — когда правила можно выразить детерминированно.  ￼

Instructor — это уже мост между LLM и этой строгой схемой. Он использует Pydantic-модели как response_model: схема одновременно задает ожидаемую структуру ответа для модели, валидирует ответ API и возвращает уже готовый Pydantic-объект. В документации прямо указано, что docstring’и и описания полей участвуют в формировании промпта, а при провале валидации Instructor умеет делать retry, чтобы добиться корректного структурированного результата.  ￼

Поэтому для анализа текста связка обычно выглядит так: Instructor извлекает из текста формальную структуру, а Pydantic проверяет жесткие ограничения. Например, из ревью-документа можно вынуть claims, evidence, risks, missing_assumptions, tone, policy_violations, а затем Pydantic проверит, что каждое утверждение имеет источник, что риск имеет severity, что финальный verdict не противоречит обязательным полям, и так далее. Это отличный подход для editorial QA, policy-checking, extraction, scoring rubric, review pipelines и подготовки “машиночитаемых” артефактов из свободного текста.  ￼

Но тут важна граница. Ни Pydantic, ни Instructor сами по себе не являются движками формальной логики: они не доказывают теоремы и не гарантируют логическую непротиворечивость текста в смысле логики предикатов. Pydantic проверяет типы и ограничения; Instructor помогает LLM вернуть данные в этой форме и, при необходимости, прогоняет повторные попытки. Для по-настоящему строгой логики их обычно используют как фронт-энд слой нормализации, а уже затем передают результат в отдельный механизм — например, SAT/SMT-решатель, rule engine, Prolog/Datalog, или ваш собственный слой правил. Это следует из самих заявленных возможностей: Pydantic про validation/schema, Instructor — про structured outputs/validation/retries.  ￼

Отдельно интересно, что Instructor уже двигается и в сторону семантической валидации: в их документации описан llm_validator, где LLM оценивает текст по сложным, контекстным, плохо формализуемым критериям вроде “профессионально ли написано”, “уважительный ли тон”, “family-friendly ли контент”. Это очень полезно для реальных текстов, потому что многие важные требования не сводятся к regex и диапазонам. Но это уже не строгая логика, а LLM-as-judge поверх структурированной схемы, то есть сильный практический компромисс, а не математическая гарантия.  ￼

На практике я бы рекомендовал смотреть на них так. Pydantic — это слой формального контракта. Instructor — это адаптер, который заставляет LLM жить внутри этого контракта. Для проверки текстов это очень мощная связка, если делать процесс в три ступени: сначала извлечь структуру, потом прогнать жесткие правила, потом при необходимости добавить семантическую проверку. Для “настоящей” формальной логики поверх этого лучше добавить отдельный логический движок.  ￼

Минимально это выглядит так:

```python
from typing import Literal
from pydantic import BaseModel, Field, field_validator, model_validator
import instructor

class Claim(BaseModel):
    text: str = Field(min_length=10)
    evidence: list[str] = Field(min_length=1)
    confidence: float = Field(ge=0.0, le=1.0)

class ReviewResult(BaseModel):
    verdict: Literal["ok", "needs_revision", "reject"]
    tone: Literal["neutral", "professional", "aggressive"]
    claims: list[Claim] = Field(min_length=1)
    contradictions: list[str] = []

    @model_validator(mode="after")
    def no_ok_with_contradictions(self):
        if self.verdict == "ok" and self.contradictions:
            raise ValueError("verdict=ok is incompatible with contradictions")
        return self

client = instructor.from_provider("openai/gpt-4o-mini")

result = client.create(
    response_model=ReviewResult,
    messages=[
        {
            "role": "user",
            "content": "Проверь этот текст, выдели утверждения, доказательства, противоречия и дай вердикт..."
        }
    ],
)
```
Такой паттерн особенно хорош, если тебе нужно не просто “мнение модели”, а контролируемый, проверяемый промежуточный артефакт, который можно дальше использовать в пайплайне.

Могу следующим сообщением показать уже более предметно: либо пример проверки статьи/PRD на противоречия, либо пример проверки пользовательского текста через Pydantic + Instructor + логические правила.

---

У меня есть агент, который генерирует спецификации по схеме, которая довольно строгая. Подойдет для такого?

---

Да, для такого кейса это очень подходит. Связка обычно выглядит так: Pydantic задает каноническую схему спецификации и проверяет инварианты, а Instructor заставляет LLM вернуть ответ именно в этой форме и умеет автоматически повторять запрос, если ответ не прошел валидацию. Это буквально тот случай, когда у тебя есть агент, генерирующий строго типизированные спецификации.  ￼

Для твоего сценария я бы разделил ответственность так. Pydantic — это “закон”: обязательные поля, enum-значения, ограничения длины, числовые диапазоны, вложенные секции, запрет пустых списков, а также межполевые проверки через field_validator и model_validator. Pydantic еще и умеет генерировать JSON Schema, что удобно, если ты хочешь сделать схему одновременно машинной спецификацией и контрактом для агента.  ￼

Instructor здесь полезен, если агент получает текстовый prompt и отвечает через LLM API. Тогда ты передаешь response_model=YourSpecModel, и библиотека использует эту Pydantic-модель как контракт ответа; если ответ невалиден, Instructor может отдать модели контекст ошибки и попробовать еще раз через max_retries. Это сильно снижает количество “почти правильных” ответов, где структура вроде похожа, но на самом деле ломает пайплайн.  ￼

Но есть важная граница: для строгой структурной корректности — да, отлично; для настоящей логической непротиворечивости содержания — только частично. Pydantic прекрасно проверит, что у спецификации есть assumptions, constraints, acceptance_criteria, что они не пустые и что priority входит в допустимый набор. Но если ты хочешь проверять вещи уровня “раздел Risks противоречит разделу Guarantees” или “acceptance criteria логически несовместимы с non-goals”, это уже лучше делать отдельным правиловым слоем поверх модели, а не надеяться только на типовую валидацию. Pydantic это допускает через кастомные валидаторы, но сложную семантическую логику удобнее выносить в отдельные функции или rule engine.  ￼

Практически я бы рекомендовал такой контур. Сначала агент генерирует не “свободный markdown”, а объект по Pydantic-схеме. Потом Pydantic прогоняет детерминированные проверки. Потом, если нужно, ты делаешь второй проход: semantic/rule validation на противоречия, дубли, пропуски обязательных смыслов. И только после этого рендеришь итоговую спецификацию в Markdown, YAML или whatever. Такой пайплайн обычно надежнее, чем пытаться валидировать уже готовый человеческий текст.  ￼

Если упростить до выбора:
	•	Если у тебя агент уже пишет JSON/dict по контракту, тебе может хватить одного Pydantic.
	•	Если у тебя агент — это LLM, который иногда “плывет” по формату, тогда Pydantic + Instructor — очень хороший вариант.  ￼

Минимальный каркас может быть таким:

```python
from typing import Literal
from pydantic import BaseModel, Field, model_validator

class AcceptanceCriterion(BaseModel):
    id: str
    text: str = Field(min_length=10)

class Specification(BaseModel):
    title: str = Field(min_length=3)
    status: Literal["draft", "ready", "approved"]
    summary: str = Field(min_length=30)
    assumptions: list[str] = Field(min_length=1)
    non_goals: list[str] = Field(min_length=1)
    acceptance_criteria: list[AcceptanceCriterion] = Field(min_length=1)

    @model_validator(mode="after")
    def check_summary_vs_title(self):
        if self.title.strip().lower() in self.summary.strip().lower():
            return self
        return self
```

А с Instructor это обычно выглядит так:

```python
import instructor

client = instructor.from_provider("openai/gpt-4o")

spec = client.create(
    response_model=Specification,
    max_retries=2,
    messages=[
        {
            "role": "user",
            "content": "Generate a specification for ..."
        }
    ],
)
```
Для твоей задачи я бы сказал так: да, подойдет, и довольно хорошо, особенно если тебе нужен не просто текст спеки, а контролируемый, валидируемый, пригодный для дальнейшей автоматизации артефакт.

Могу дальше показать уже более прикладной вариант именно под твой формат: например, набросать Pydantic-схему для спецификации уровня PRD/OpenSpec/SpecGraph с кросс-проверками вроде completeness, traceability и contradiction guards.
