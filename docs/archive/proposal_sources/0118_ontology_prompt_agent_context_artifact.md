# Ontology Prompt-Agent Context Artifact

## Source Intent

After `0117`, ordinary supervisor runs can see ontology semantic gate evidence,
but the next generation boundary is still implicit. Before a supervisor agent
or SpecSpace shell drafts graph-facing text, it should receive a typed ontology
context that narrows terminology to accepted project vocabulary and makes
aliases, deprecated terms, relation conflicts, unresolved gaps, package refs,
versions, digests, and evidence refs explicit.

This is not a strict ontology compiler. It is a grounding artifact for MVP
prompt-agent use so the agent has less room to invent terms or use names that
the project ontology already rejected.

## Requested Work

- Upgrade `runs/ontology_prompt_invocation_index.json` from an empty placeholder
  into a typed prompt-agent ontology context boundary.
- Use the existing `ontology_semantic_context_pack` as source context.
- Carry package refs, versions, digests, accepted terms, accepted relations,
  aliases, deprecated terms, relation conflicts, unresolved gaps, governance
  evidence, prompt input refs, prompt output refs, and failure modes.
- Keep the artifact context-only: do not execute prompt agents, persist raw
  prompts, persist raw responses, write Ontology packages, update lockfiles,
  mutate canonical SpecGraph specs, mark candidates accepted, or close semantic
  gates.
- Make the artifact deterministic and generated by `make ontology-imports`.

## Follow-Up Shape

The next slice should move to the Ontology repository and produce real
owner-reviewed decisions for delta candidates. SpecGraph can already preview
accepted/rejected decisions, but the current fixture is still graph-side test
data rather than an Ontology-owned decision stream.
